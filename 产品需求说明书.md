# A股复盘截图-OCR-AI结构化入库工具 系统功能需求文档

## 1. 文档目的

本文档用于明确一套基于 **PySide6 + Python + Umi-OCR + SQLite** 的 Windows 常驻托盘工具的系统功能需求。该工具面向 A 股复盘场景，支持用户从多个页面进行截图，调用本地 Umi-OCR 接口完成 OCR 识别，再结合为不同 `CaptureType` 配置的 PromptTemplate，将 OCR 结果发送给指定 AI 模型进行结构化分析，最终以 JSON 形式预览、修改并入库。文中所有CaptureType，在应用程序界面中指的是`业务类型`，显示文字也采用`业务类型`。

本文档聚焦：
- 产品边界
- 交互流程
- 功能模块
- 数据模型
- 状态约束
- 异常处理
- 非功能需求
- 开发阶段建议

---

## 2. 系统目标

### 2.1 业务目标

构建一个适用于 Windows 的常驻型桌面工具，服务于 A 股复盘、市场概览、情绪分析、题材跟踪等场景，实现：

1. 用户按不同业务类型发起截图。
2. 每种截图类型绑定独立 PromptTemplate。
3. 截图后直接调用本地 Umi-OCR 接口识别，不进行图片预处理。
4. OCR 结果与对应 Prompt 一起发送给指定 AI 模型。
5. AI 返回结构化 JSON 数据。
6. 用户可预览与修改结构化结果。
7. 用户确认后按“日期 + CaptureType”唯一键入库；同一天同一类型多次解析时，后写覆盖前写。

### 2.2 产品定位

该工具是一个 **本地优先、本地 OCR、云端或本地 AI 结构化分析、最终结构化入库** 的桌面应用，不是通用 OCR 工具，也不是通用截图工具，而是围绕“复盘数据采集与结构化沉淀”设计的专用工作台。

## 3. 技术范围与约束

### 3.1 技术栈

- 桌面 UI：PySide6
- 运行语言：Python 3.x
- OCR：本地 Umi-OCR HTTP 接口
- 数据库：SQLite
- 网络请求：requests 或 httpx
- JSON 校验：json / pydantic / jsonschema（二期可加入）

### 3.2 明确约束

1. 平台仅考虑 Windows。
2. 工具以 **系统托盘常驻** 方式运行。
3. 托盘菜单仅保留三个一级入口：
   - 截图
   - 设置
   - 退出
4. 截图后 **不做任何图片预处理**，直接发送到 Umi-OCR 接口。
5. OCR 接口先于 AI 调用执行。
6. AI 请求可能耗时较长，必须支持：
   - 防重复点击
   - 动态处理中状态
   - 请求失败后的恢复
7. 入库按“指定日期 + CaptureType”唯一保存。
8. 同一天、同一 CaptureType 多次确认入库时，执行覆盖写入。

## 5. 核心业务概念

### 5.1 CaptureType

`CaptureType` 表示截图业务类型，例如：
- 市场动态
- 市场总览
- 涨停复盘
- 连板高度
- 题材强度
- 北向资金

每个 CaptureType 绑定：
- 唯一名称
- 描述（可选）
- PromptTemplate
- 返回 JSON 结构说明
- 是否启用
- 创建时间 / 更新时间

### 5.2 PromptTemplate

PromptTemplate 为某一 CaptureType 的分析模板。它应明确告诉 AI：
- 输入内容来源于 OCR 文本
- 需要提取哪些字段
- JSON 返回格式是什么
- 禁止输出解释性文字，仅返回 JSON 或严格结构化内容

### 5.3 AI 供应商

AI 供应商表示一个可调用的模型服务来源，例如：
- OpenAI 兼容接口
- DeepSeek
- 通义千问兼容服务
- 本地模型代理服务
- 其他自定义兼容 OpenAI 格式服务

每个供应商支持：
- 供应商名称
- API Base URL
- API Key
- 多个模型配置
- 默认模型设置
- 启用/禁用状态

### 5.4 解析结果

一次完整解析流程的产物包括：
- 截图文件
- OCR 原始文本
- AI 输入内容
- AI 输出 JSON 文本
- 用户最终确认后的 JSON 文本
- 对应日期
- 对应 CaptureType

## 6. 总体业务流程

### 6.1 配置阶段

1. 用户通过“设置”进入配置界面。
2. 在 Tab1 中维护 CaptureType 和 PromptTemplate。
3. 在 Tab2 中维护 AI 供应商、API 地址、Key、模型列表及默认模型。
4. 配置保存到 SQLite。

### 6.2 使用阶段

1. 用户点击托盘菜单“截图”。
2. 弹出 CaptureType 列表。
3. 用户点击某个 CaptureType。
4. 进入自由截图模式（用户按住鼠标左键点选A点，拖动鼠标到B点，点击后截图，拖动过程显示方框，表示截图区域，按ESC键取消截图）。
5. 用户完成截图后显示预览框。
6. 预览框提供：
   - 重新截取
   - 发送解析
7. 用户点击“发送解析”。
8. 系统先调用本地 Umi-OCR 接口，获得 OCR 文本。
9. 系统读取当前 CaptureType 绑定的 PromptTemplate。
10. 系统读取当前生效的 AI 供应商与模型配置。
11. 系统将 OCR 文本 + Prompt 发送给 AI。
12. 请求期间 UI 按钮禁用并显示处理中状态。
13. 收到 AI 返回后，展示结构化结果预览。
14. 用户可修改结构化结果。
15. 用户选择日期，（默认当前日期）并确认“入库”。
16. 系统按“日期 + CaptureType”写入 SQLite。
17. 若该唯一键已存在，则覆盖更新。

## 7.2 设置窗口

设置窗口分为两个 Tab。

### 7.2.1 Tab1：CaptureType 管理

该 Tab 用于管理业务截图类型及其 PromptTemplate。

#### 页面组成
- 左侧：CaptureType（中文业务类型） 列表
- 右侧：详情编辑区
- 底部或顶部：新增、保存、删除

#### 列表显示字段
- CaptureType 名称
- 是否启用
- 更新时间

#### 支持功能

##### 1）新增 CaptureType
用户可新增一个业务类型，例如“市场动态”。

新增时至少需要填写：
- CaptureType 名称
- PromptTemplate

可选字段：
- 描述
- 是否启用

##### 2）编辑 CaptureType
用户可修改已有 CaptureType 的：
- 名称
- 描述
- PromptTemplate
- 是否启用

##### 3）PromptTemplate 编辑要求
PromptTemplate 编辑区需要支持输入较长文本，建议使用多行文本框。

用户在模板中应能写入：
- 分析说明
- 目标字段
- JSON 返回格式约束

例如要求 AI 返回：

{
  "trade_date": "2026-04-17",
  "market_sentiment": "strong",

- 校验名称不能为空
- 校验 PromptTemplate 不能为空

#### 支持功能

##### 1）新增供应商

支持修改：
- 名称
- API Base URL
- API Key
- 默认模型

每个供应商支持多个模型，用户可：
- 新增模型
- 编辑模型
- 禁用模型
- 设为默认模型

##### 4）默认模型规则
- 每个供应商只能有一个默认模型。
- 若用户未在解析时临时指定模型，则使用当前供应商默认模型。

##### 5）默认供应商规则（建议）

## 7.4 OCR 识别流程

### 7.4.1 OCR 调用顺序

点击“发送解析”后，系统必须先调用本地 Umi-OCR 接口。

### 7.4.2 OCR 输入要求
- 输入为截图生成的图片文件。
- 不进行图像预处理。
- 直接将图片发送到 Umi-OCR 接口。

### 7.4.3 OCR 输出要求
OCR 成功后至少得到：
- OCR 原始文本

若接口可返回更多字段，二期可扩展保存：
- 行级文本
- 坐标信息
- 置信度

### 7.4.4 现有脚本基础

用户已提供一个 Python 脚本，已具备如下基础能力：
- 读取本地图片文件
- 转 base64
- 通过 `POST {umi_url}/api/ocr` 调用 Umi-OCR 接口
- 使用 `data.format = text`
- 使用 `tbpu.parser = single_line`
- 在响应 `code == 100` 时返回 OCR 文本

该脚本说明当前 OCR 集成方向已经明确，可作为 OCR 服务层封装的初始基础。fileciteturn0file0

### 7.4.5 OCR 失败处理
当 OCR 调用失败时，系统应：
- 停止后续 AI 请求
- 提示错误原因
- 恢复按钮可点击状态
- 允许用户重新发送解析

错误场景包括但不限于：
- Umi-OCR 服务未启动
- 接口连接超时
- 返回码异常
- 返回内容为空

## 7.5 AI 结构化分析流程

### 7.5.1 输入组成
AI 请求输入由以下部分组成：
1. 当前 CaptureType 对应 PromptTemplate
2. OCR 原始文本
3. 必要的系统提示（如“仅返回 JSON，不要输出解释”）

### 7.5.2 模型选择规则
默认使用：
- 当前默认供应商
- 当前默认模型

若后续扩展，也可支持在发送前切换本次使用的供应商与模型，但不是一期硬需求。

### 7.5.3 请求中状态管理
AI 请求可能较慢，因此必须满足以下要求：

#### 按钮防重入
点击“发送解析”后：
- 按钮立即置灰或禁用
- 不允许重复点击
- OCR 与 AI 整个处理链路完成前不得恢复

#### 动态反馈
请求处理中时，应显示明显的动态状态，例如：
- 按钮文字切换为“解析中...”
- 显示 loading 动画 / 旋转图标
- 显示阶段提示：
  - OCR识别中
  - AI分析中

#### 界面不卡死
AI 请求应在后台线程执行，不能阻塞主线程。
建议：
- 使用 `QThread`、`QRunnable + QThreadPool` 或异步任务封装
- 用信号槽回传状态更新

### 7.5.4 AI 返回要求
AI 返回的核心目标是结构化 JSON 文本。

系统至少需要拿到：
- 原始返回字符串

建议二期增加：
- JSON 解析校验
- 字段合法性校验
- 自动格式化展示

### 7.5.5 AI 异常处理
当 AI 请求失败时，系统应：
- 展示失败提示
- 保留截图与 OCR 结果
- 恢复发送解析按钮
- 允许重新发送

错误场景包括：
- API Base URL 配置错误
- Key 无效
- 网络超时
- 模型不存在
- 返回非 JSON
- 返回为空

## 7.6 结果预览与修改

### 7.6.1 结果展示界面
AI 返回后，系统应弹出或切换到结果确认界面。

#### 必须展示内容
- 当前 CaptureType
- 关联日期（默认当天，可修改）
- OCR 原始文本，可折叠显示
- AI 返回结果

### 7.6.2 结果编辑能力
用户必须能对 AI 返回结果进行手工修改。

#### 编辑要求
- 支持多行文本编辑
- 建议进行 JSON 格式化展示
- 用户可直接修改字段值
- 支持json格式化验证
- 若文本不是合法 JSON，入库前应提示

### 7.6.3 日期选择
用户确认入库前，应支持选择结果所属日期，默认当前日期。

#### 默认值
- 默认当天日期

#### 用途
该日期与 CaptureType 组成唯一业务键，用于覆盖写入。

### 7.6.4 入库按钮
结果确认界面应提供“入库”按钮。

点击后：
- 校验当前结果非空
- 校验 JSON 合法性
- 校验日期非空
- 执行数据库写入

## 7.7 入库规则

### 7.7.1 唯一规则
业务上，每个：
- 日期
- CaptureType

仅允许存在一条最终确认结果。

即唯一键为：
- `result_date + capture_type_id`

### 7.7.2 覆盖规则
若同一天同一 CaptureType 再次入库：
- 不新增多条
- 而是覆盖之前的记录

### 7.7.3 建议保留历史字段
虽然主业务记录只保留一条最终结果，但建议保留以下审计字段：
- created_at
- updated_at
- ocr_text
- ai_raw_response
- final_json_text
- image_path

若二期考虑审计追踪，可加入单独日志表记录每次解析尝试。

- 支持编辑
- 支持日期选择
- 支持入库按钮

建议增加：
- JSON 格式化按钮
- 复制结果按钮
- 校验 JSON 按钮

## 10. 状态机需求

为了避免混乱，建议明确界面状态。

### 10.1 截图任务状态

- `idle`：空闲
- `capturing`：截图中
- `previewing`：预览中
- `ocr_processing`：OCR处理中
- `ai_processing`：AI处理中
- `result_editing`：结果编辑中
- `saving`：入库中
- `done`：完成
- `failed`：失败

### 10.2 状态约束

#### 空闲态
- 可开始截图
- 不存在处理中任务

#### OCR处理中
- 禁止再次点击发送解析
- 禁止重复创建相同任务

#### AI处理中
- 同样禁止再次触发发送解析
- 应保留取消能力（一期可不做取消，只做禁点）

#### 结果编辑中
- 允许编辑文本
- 允许修改日期
- 允许入库

## 11. 异常与边界场景

## 11.1 配置类异常
- 没有任何启用的 CaptureType
- 没有任何启用的 AI 供应商
- 供应商存在但没有默认模型
- PromptTemplate 为空

处理建议：
- 给出明确提示，不进入后续流程

## 11.2 截图类异常
- 用户取消截图
- 截图结果为空
- 图片保存失败

处理建议：
- 安静返回或提示重试

## 11.3 OCR 类异常
- Umi-OCR 未启动
- URL 配置错误
- OCR 返回空字符串

处理建议：
- 提示错误并恢复到可重试状态

## 11.4 AI 类异常
- 网络超时
- 401/403
- 模型不存在
- 返回非 JSON

处理建议：
- 错误提示
- 允许重试
- 不自动入库

## 11.5 入库类异常
- SQLite 文件被占用
- JSON 非法
- 唯一键冲突更新失败

处理建议：
- 明确提示
- 保留用户编辑内容，避免丢失

## 12. 非功能需求

## 12.1 性能
- 普通截图预览应在 1 秒内完成。
- OCR 请求应支持超时设置。
- AI 请求应支持较长等待时间和超时配置。

## 12.2 可用性
- 常用操作路径短。
- 截图到解析确认流程尽量不超过 3 个主界面切换。
- 错误提示应可读、明确、可操作。

## 12.3 可靠性
- SQLite 写入要具备事务性。
- 程序崩溃前已保存配置不应丢失。
- 临时文件建议按规则清理。

## 12.4 安全性
- API Key 不应明文暴露在 UI 列表中。
- 建议支持“显示/隐藏”Key。
- 二期建议对 Key 做本地加密存储。

## 12.5 可维护性
- OCR 服务层、AI 服务层、数据库层、UI 层分离。
- CaptureType 与 PromptTemplate 配置化。
- 供应商适配层可扩展。

## 16. 验收标准

当满足以下条件时，可判定一期需求基本达成：

1. 程序可在 Windows 托盘稳定常驻。
2. 托盘菜单仅含截图、设置、退出三个入口。
3. 可在设置中新增、编辑、保存 CaptureType 与 PromptTemplate。
4. 可在设置中新增多个 AI 供应商，并为每个供应商维护多个模型及默认模型。
5. 用户可从截图入口选择某个 CaptureType 并自由截图。
6. 截图后可预览，并能重新截取或发送解析。
7. 发送解析后系统先调用 Umi-OCR，再调用 AI。
8. 处理过程中按钮不会被重复点击，且有动态处理中反馈。
9. AI 返回后可预览和修改结果。
10. 用户确认后可按日期入库。
11. 同一日期 + CaptureType 再次入库时会覆盖旧记录。
12. 主要异常场景下程序不会崩溃，并能提示用户重试。

---

## 17. 结论

本需求的一期重点不是做复杂自动化，而是先做出一条稳定、清晰、可人工确认的链路：

**CaptureType 配置 → 截图 → Umi-OCR → Prompt + OCR 发给 AI → JSON 结果确认 → 覆盖入库**

只要这一主链路稳定，后续再扩展历史查询、批量解析、日志追踪、导出报表，成本会低很多。
